# DIANA_STANDARD.md

This file is the single source of truth for Diana's quality standard. Every specialist in this system references it. If the standard changes, it changes here — not in individual folders.

---

## The Standard

Diana's team turns down volume to do better work. That choice shows up in every client interaction. The standard has four criteria, applied in this order:

**1. Specificity over generality**
Every output — draft, research brief, lead profile, checklist — should reflect the specific client, property, and situation it was made for. If it could have been produced for a different client with the same effort, it isn't specific enough.

**2. Clarity over comfort**
Hard information is delivered directly. Bad news, data that contradicts expectations, risks the client needs to know about — these are named plainly, not cushioned into ambiguity. Clients trust this team because they get real information. Softening information to manage short-term discomfort erodes that trust over time.

**3. Brevity**
No filler. No padding. No throat-clearing. Every sentence in every output should be doing work. If removing a sentence doesn't change what the reader understands or needs to do, remove it.

**4. Voice**
Client-facing communication sounds like the agent who signs it — not like a system, not like a template. The voice folder exists to make this concrete. A message that could have been sent by anyone on the team, or by no one in particular, has failed this criterion.

---

## What the Standard Is Not

- It is not a style guide. It describes judgment, not formatting.
- It is not a checklist to run through after the work is done. It should shape the work as it's being produced.
- It is not the same for every specialist. Each folder applies the standard to its own domain — see the Diana Standard section in each specialist's `rules.md` for what it means there specifically.

---

## Escalate to Diana When

Any output that would make Diana say "I'd want to have seen this before it went out" should come to her first. Each specialist defines its own escalation triggers in `rules.md`. The common thread across all of them:

- The stakes are high enough that a mistake is hard to recover from
- The situation is outside the team's standard experience
- The output is borderline — not clearly wrong, but Diana's read would add value
- A client relationship that matters especially to Diana is involved

---

## Updating This Standard

This file should be updated when Diana's standards genuinely evolve — not to document every preference, but when the team has learned something that should change how everyone operates. Changes here ripple through the whole system. Make them deliberately.
